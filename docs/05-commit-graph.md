# 05. 提交图绘制

提交图通过纵向“泳道”（Lane）展示 Git 仓库的拓扑结构（DAG）。系统采用高效的行列计算算法，确保在虚拟滚动环境下依然能保持流畅的渲染性能。

## 核心目标

- **拓扑还原**：准确展示主线、分支、合并（Merge）及樱桃拣选（Cherry-pick）的演进关系。
- **无状态渲染**：每一行的图形数据（Segments）仅依赖于后端提供的提交信息，实现与虚拟滚动的完美兼容。
- **视觉区分**：对丢失引用的提交（Unreachable）和贮藏（Stash）进行特殊的样式标记。
- **风格自定义**：支持多种线条风格（圆润、直角、斜线），满足不同用户的审美偏好。

## 算法实现策略

算法在前端对日志数据进行单次扫描，动态维护泳道状态：

1. **泳道分配**：为每个提交及其父提交分配唯一的列号。优先复用已有泳道以保持主线的直观性。
2. **状态快照**：记录每一行“进入”和“离开”时的泳道布局，用于计算连接线段。
3. **线段生成**：
   - **直连线**：同一分支在连续行间的垂直连接。
   - **分支/合并线**：通过斜线或折线表示分支的开启与合并汇入。
   - **颜色管理**：为新开启的泳道分配循环色，确保视觉上的分支辨识度。

## 渲染机制

- **SVG 驱动**：使用 SVG 矢量图形进行高效率渲染。
- **动态样式**：根据全局设置计算路径参数，支持 `rounded`（圆润）、`step`（直角步进）和 `angular`（折线）三种几何风格。
- **列宽自适应**：根据当前可见范围内最大的泳道数量动态调整图形区的宽度。

## 特殊场景处理

### 搜索模式
由于搜索结果通常是不连续的，提交图在搜索生效时会自动隐藏，以避免展示错误的拓扑关系。

### 丢失引用与贮藏
- **Unreachable**：通过特殊的颜色减淡和线框样式标记，表示该提交已不在常规的分支引用链中。
- **Stash**：在算法层面将其视为特殊的单父提交，并在渲染时使用空心图标进行区分。

### 数据预处理
后端在提供日志数据时，会预先剔除 Git 内部产生的辅助提交（如 Stash 的 index/untracked 快照），并对 Parent 列表进行必要修剪，确保前端算法能将其视为标准的 DAG 结构进行处理。

## 关键技术决策

1. **局部依赖性**：通过将复杂的图算法计算结果转化为每行独立的线段数据，极大降低了长列表滚动的计算负担。
2. **稳定配色方案**：泳道颜色随其“首次出现”顺序递增，确保在同一视图会话中分支颜色的稳定性。
3. **简化 Stash 结构**：在不改变 Git 物理存储的前提下，逻辑上简化 Stash 的父子关系，使其更符合用户对“提交历史”的直观理解。
