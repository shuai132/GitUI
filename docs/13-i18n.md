# 13. 国际化 (i18n)

GitUI 支持多语言界面，致力于为全球开发者提供无障碍的操作体验。

## 核心原则

- **标准驱动**：基于主流的国际化库构建，确保翻译资源的组织与加载符合工程化标准。
- **自动适配**：默认采用“跟随系统”策略，同时允许用户在设置中显式指定语言。
- **即时响应**：语言切换在应用运行时即时生效，无需重启进程。
- **术语一致性**：核心 Git 术语及品牌名在不同语言环境下保持行业通用表达。

## 实现逻辑

### 语言解析与探测
系统在启动时会根据以下优先级自动探测语言：
1. 用户显式指定的持久化偏好。
2. 应用运行环境的系统语言设置。
3. 默认兜底语言。

为了优化启动性能，系统会缓存探测结果，以确保二次启动时能够同步应用正确的主题。

### 资源组织
翻译资源按功能域进行结构化组织。每一组文案都拥有唯一的键名（Key），支持命名插值以处理动态内容。

### 错误消息映射
由于底层 Git 引擎返回的错误消息可能包含多语言混合内容，系统建立了专门的映射机制：
- **模式识别**：通过对原始错误消息进行关键字识别，将其映射为标准的国际化键名。
- **友好提示**：将晦涩的底层错误转换为用户易于理解的业务描述。

## 开发规范

### 文案引用
- **组件内**：通过标准插件提供的函数进行引用，支持参数注入。
- **业务逻辑层**：对于独立于 UI 的业务代码，通过全局实例进行文案渲染。

### 品牌名处理
系统保留行业通用的英文名称（如分支名、Git 操作术语、特定的字体或编辑器名称）不进行强制翻译，以降低认知偏差。

### 国际化覆盖范围
- **界面静态文案**：菜单、标签、提示文字。
- **动态业务反馈**：操作成功/失败通知。
- **错误代码映射**：详尽的 Git 错误解释。

## 关键技术决策

1. **按需刷新**：语言切换会触发根级状态更新，确保所有已挂载组件同步刷新。
2. **前后端解耦**：国际化逻辑完全由前端驱动，后端仅负责提供结构化的原始错误数据，由前端负责最终的本地化呈现。
