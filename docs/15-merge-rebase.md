# 15. 合并与变基 (Merge / Rebase)

本模块提供完整的合并、变基及冲突解决工具链，支持标准的版本集成流与交互式历史修整。

## 核心职责

- **合并 (Merge)**：支持快进合并（Fast-forward）、普通合并（Non-FF）及压缩合并（Squash）。
- **变基 (Rebase)**：支持线性的非交互式变基，以及包含修补（Reword）、压缩（Squash/Fixup）、丢弃（Drop）和重排（Reorder）的交互式变基。
- **冲突解决**：提供集成的三路合并编辑器，支持按行选择冲突内容。
- **中间态管理**：实时感知仓库的中间状态（Merge/Rebase/Cherry-pick/Revert 中），并提供相应的操作横幅。

## 入口与交互流

### 触发机制
- **上下文菜单**：从历史视图的提交行触发合并、变基或消息修补（Reword）。
- **拖拽交互**：支持通过提交间的拖拽快速发起操作，并弹出动作选择对话框。
- **操作横幅 (Banner)**：当仓库处于中间态时，全局横幅提供“继续”、“跳过”或“中止”的便捷入口。

### 冲突解决工作流
1. **感知**：冲突文件以特殊状态出现在工作区列表中。
2. **编辑**：通过三路合并编辑器展示“当前分支（Ours）”、“目标分支（Theirs）”及“基础版本（Base）”。
3. **决策**：用户按行勾选需要保留的内容，系统实时合成输出预览。
4. **完结**：保存合成内容并标记为已解决，通过索引操作替换冲突条目。

## 数据集成逻辑

### 状态透传
后端在执行状态查询（Status）时，会同步采集仓库的中间态信息（如合并消息、变基进度等），并将其封装在 `RepoState` 中透传给前端 Store 及 UI 横幅。

### 变基计划 (Rebase Plan)
交互式变基在启动前会生成一个计划清单，允许用户在前端调整各提交的动作（Pick/Reword/Squash 等）及顺序。这些动作会在执行过程中由后端循环调度。

### 自动贮藏 (Autostash)
针对脏工作区场景，系统提供可选的自动贮藏与恢复机制，确保集成操作在干净的环境下进行，并在完成后尝试恢复用户的本地改动。

## 关键技术决策

1. **按行粒度合并**：合并编辑器优先保证行级冲突的选择效率，不支持在编辑器内直接进行文本编辑，以确保冲突解决的一致性。
2. **时间戳保留与修补**：在 Reword 或修补历史时，支持自定义作者与提交者时间，以满足精确的历史回溯需求。
3. **状态持久化**：变基计划等中间数据通过仓库内的临时文件进行持久化，确保在应用重启后仍能继续之前的操作流。
4. **错误兜底**：对于自动恢复（如 Autostash Pop）失败的场景，系统采取保守策略，仅提示用户手动处理，避免造成二次代码丢失。

## 刷新与同步

任何集成操作的推进或终止都会自动触发全局状态（历史、分支、工作区）的刷新，确保用户始终看到最准确的版本拓扑。
